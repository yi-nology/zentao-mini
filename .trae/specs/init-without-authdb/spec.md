# 禅道 Mini 初始化改进 - 产品需求文档

## 概述
- **摘要**: 改进禅道 Mini 后端的初始化流程，使其在 auth.db 数据库不存在时也能正常初始化，同时确保用户进行强初始化后能默认加载禅道 SDK，并且程序重启后能正常加载配置。
- **目的**: 解决 auth.db 不存在时的初始化问题，提供更灵活的启动方式，确保禅道 SDK 能正确加载。
- **目标用户**: 禅道 Mini 应用的开发者和使用者。

## 目标
- 当 auth.db 数据库不存在时，程序仍能正常初始化启动
- 实现用户强初始化机制，确保配置正确设置
- 默认加载禅道 SDK，确保功能正常运行
- 程序重启后能正常加载配置，无需重新初始化

## 非目标（范围外）
- 不修改现有的加密配置机制
- 不改变现有的数据库存储结构
- 不影响已有的禅道 SDK 功能

## 背景与上下文
- 当前实现中，当 auth.db 不存在时，程序会尝试加载加密配置文件，如果失败则依赖环境变量
- 如果环境变量也未设置，程序会直接退出
- 用户希望即使在 auth.db 不存在的情况下，程序也能启动并提供初始化机制

## 功能需求
- **FR-1**: 当 auth.db 不存在时，程序应能正常启动，进入初始化模式
- **FR-2**: 实现强初始化机制，要求用户提供必要的配置信息
- **FR-3**: 初始化成功后，默认加载禅道 SDK
- **FR-4**: 程序重启后，应能正常从 auth.db 加载配置

## 非功能需求
- **NFR-1**: 初始化过程应简洁明了，引导用户完成必要配置
- **NFR-2**: 配置存储应安全可靠，沿用现有的加密机制
- **NFR-3**: 程序启动时间不应显著增加

## 约束
- **技术**: Go 语言实现，沿用现有的初始化服务架构
- **依赖**: 现有的禅道 SDK 和加密配置机制

## 假设
- 用户会在初始化过程中提供正确的禅道服务器配置
- 程序运行环境具有文件系统写入权限

## 验收标准

### AC-1: 无 auth.db 时程序启动
- **给定**: auth.db 文件不存在
- **当**: 启动程序
- **则**: 程序正常启动，进入初始化模式
- **验证**: `programmatic`

### AC-2: 强初始化机制
- **给定**: 程序进入初始化模式
- **当**: 用户提供必要的配置信息
- **则**: 配置被存储到 auth.db，程序继续运行
- **验证**: `programmatic`

### AC-3: 禅道 SDK 加载
- **给定**: 初始化成功
- **当**: 程序启动
- **则**: 禅道 SDK 被默认加载，功能可用
- **验证**: `programmatic`

### AC-4: 重启后正常加载
- **给定**: 初始化成功并存储配置
- **当**: 重启程序
- **则**: 程序从 auth.db 正常加载配置，无需重新初始化
- **验证**: `programmatic`
